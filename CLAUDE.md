# CLAUDE.md — 소울유니버스 코딩 테스트 작업 규칙

이 저장소는 채용 코딩 테스트다. **평가의 핵심은 "왜 그렇게 결정했는가"** 이며,
정답이 하나가 아니므로 **판단과 트레이드오프 근거**가 점수다. 모든 작업은 이 원칙을 따른다.

## 과제 구성

- **Part A** — Flutter 앱(`lib/**`)에 의도적으로 심어진 버그 수정. (`REPORT.md` 표 기준 약 5~6개)
- **Part B** — 관리자용 MySQL 쿼리 2개 개선. 작성 위치는 `sql/queries_b.sql`.
- 분석/플랜 보조 문서는 `docs/**`. 공식 제출물은 루트 `REPORT.md`.

## 절대 규칙 (위반 금지)

1. **`sql/queries_b.sql` 상단 "스키마 참조" 블록과 "[문제 쿼리]"는 수정하지 않는다.** 개선 쿼리는 지정된 `[개선된 쿼리]`/`[CREATE INDEX]` 자리에만 작성.
2. **`node_modules`/`build`/`.dart_tool`/`.idea`/`.vscode`는 커밋·제출에서 제외.** (`.gitignore` 준수)
3. **AI 도구(Claude Code 포함) 사용 사실과 범위를 README와 REPORT.md에 반드시 고지한다.**

## 커밋 규율

- **버그 1개 = 커밋 1개**를 기본으로 한다. (같은 파일의 "상태 갱신 누락"류는 묶을 수 있으나, 묶는 이유를 메시지에 한 줄로 남긴다.)
- 메시지는 **한국어**, `type(scope): 무엇을 — 왜` 형태로 *이유*가 드러나게.
  - 타입: `fix`(버그), `perf`(쿼리 성능), `docs`(문서), `chore`(설정/빌드)
  - 예: `fix(favorite): 즐겨찾기 삭제 조건을 == 에서 != 로 수정 — 대상만 남기고 나머지를 지우던 버그`
- 커밋 전 `flutter analyze`로 회귀 없는지 확인.
- **사용자가 명시적으로 요청할 때만 커밋한다.** (임의 커밋 금지)

## 테스트 & PR 워크플로우 (Part A 버그)

버그 수정은 **테스트 우선(red→green) + PR 단위**로 진행한다. 상세 절차는 `flutter-bugfix` 스킬.

- **버그 1개 = 브랜치 1개 = PR 1개.** 브랜치는 `main` 기준 `fix/a{번호}-{slug}`.
- **테스트 먼저 작성**해 의도된 시나리오를 재현하고 `flutter test`로 **실패(red)** 확인 → 커밋(`test: ...`).
  이어 **수정 후 통과(green)** 확인 → 커밋(`fix: ...`). 즉 "문제→해결"이 2커밋으로 이력에 남는다.
- 테스트는 `flutter test`(Chrome 불필요)로 검증. 상태 로직은 unit, UI 동작은 widget test.
- **PR은 생성만 하고 머지하지 않는다.** 머지는 사용자가 직접 리뷰 후 수행.
- PR 본문에 버그/근본원인/테스트 red→green/검증 방법을 기재.

## REPORT.md 작성 원칙

- **코드 설명이 아니라 의사결정 근거**를 쓴다. "무엇을 바꿨다"보다 "왜 이 선택이 옳은가 / 다른 대안과의 트레이드오프".
- Part A: 버그마다 표 1행(파일·수정·원인). 수정 즉시 갱신.
- Part B: 문제점 → 개선 쿼리 → 개선 이유(가능하면 EXPLAIN before/after 근거).
- 요구사항 해석이 갈리는 지점(예: "최근 30일" 경계, 완료 0건 상담사 포함 여부)은 **택한 해석과 이유**를 명시.

## 코드 스타일 (Flutter)

- 기존 코드의 관습을 따른다: `provider`(`ChangeNotifier` + `context.watch/read`), `go_router`, mock repository 패턴.
- UI 문자열·주석은 **한국어** 유지.
- 상태 변경 후에는 반드시 `setState`/`notifyListeners`로 반영. `StreamSubscription` 등 리소스는 `dispose`에서 해제.
- 최소 침습 수정: 버그를 고치되 주변 구조를 불필요하게 갈아엎지 않는다. 구조 개선이 필요하면 별도 커밋 + REPORT에 근거.

## 작업 흐름

- Part A 버그 수정 → `flutter-bugfix` 스킬 절차를 따른다.
- Part B 쿼리 개선 → `sql-optimize` 스킬 절차를 따른다.
- 상세 분석/플랜은 `docs/01~03`을 참고하고, 진행하며 갱신.

### 스킬 실행 시점 안내 (필수)

- **하나의 작업이 끝날 때마다(매 응답 마무리에)**, 다음에 **어떤 스킬을 어느 시점에 실행해야 하는지**를 사용자에게 명시적으로 알려준다.
  - 예: "다음은 A-2 버그입니다 → 착수 전 `flutter-bugfix` 스킬을 실행할 시점입니다", "Part A가 끝났으니 Part B 첫 쿼리 착수 시 `sql-optimize`를 실행하세요".
- 해당 시점에 실행할 스킬이 없으면 "지금은 실행할 스킬 없음"이라고 명시한다.
- 목적: 스킬 트리거 타이밍을 사용자가 항상 파악하고 통제할 수 있게 한다.
