# flutter-bugfix

소울유니버스 코딩 테스트 Part A 전용. **버그 1개를 끝까지(수정+검증+기록+커밋 준비) 처리**하는 절차.
평가 기준은 "왜 그렇게 고쳤는가"이므로, 수정 자체보다 **근거 기록**을 빠뜨리지 않는 게 목적이다.

## 절차

### 1. 진단 (Diagnose)
- 대상 버그의 **증상 → 원인 코드 위치(`파일:라인`) → 근본 원인**을 한 문장씩 정리.
- 정상 동작 기준은 `docs/00-overview.md`의 3개 시나리오(닉네임/즐겨찾기/채팅)와 대조.
- `docs/01-part-a-analysis.md`에 해당 항목이 있으면 신뢰도(🔴/🟡/⚪)와 대조해 확정.

### 2. 수정 (Fix) — 최소 침습
- 버그의 **근본 원인만** 고친다. 주변 리팩터링은 별도 사안으로 분리.
- 상태 변경 후 `setState`/`notifyListeners` 반영, 리소스는 `dispose`에서 해제 — CLAUDE.md 스타일 준수.
- 한 번에 **하나의 버그**만 다룬다. (같은 파일의 "상태 갱신 누락"류 묶음은 허용하되 묶는 이유를 기록)

### 3. 검증 (Verify) — 필수, 건너뛰지 말 것
- `flutter analyze`로 정적 오류·경고 회귀 없는지 확인.
- 가능하면 실행 확인: 플랫폼 파일 없으면 `flutter create . --platforms android,ios` 후 `flutter run`, 해당 시나리오를 손으로 재현.
- 실행이 불가하면 그 사실과 "코드 추론으로 검증"임을 REPORT에 솔직히 명시.

### 4. 기록 (Report)
- `REPORT.md` Part A 표에 1행 추가: `# | 파일 경로 | 수정 내용 | 원인 분석`.
- **원인 분석 칸은 "왜"가 드러나게.** 예: "삭제 조건이 `==`라 대상만 남기고 나머지를 제거 → `!=`로 의미 정정".
- 대안을 고려했다면(예: 다른 수정 방식) 트레이드오프를 "기타 의견"이나 PR 코멘트에 한 줄.

### 5. 커밋 준비 (Commit)
- **사용자가 커밋을 요청할 때만** 실행. 임의 커밋 금지.
- 형식: `fix(scope): 무엇을 — 왜` (한국어). 예:
  `fix(chat): 실시간 수신 메시지에 setState 추가 — 리스트만 갱신돼 화면 미반영되던 버그`
- 버그 1개 = 커밋 1개 원칙.

## 산출물 체크리스트 (버그 1건당)
- [ ] 원인을 `파일:라인`과 함께 한 문장으로 설명 가능
- [ ] 최소 침습 수정 완료
- [ ] `flutter analyze` 통과 (가능하면 실행 확인)
- [ ] `REPORT.md` 표 1행 + 근거 작성
- [ ] (요청 시) 규칙에 맞는 커밋 메시지 준비
