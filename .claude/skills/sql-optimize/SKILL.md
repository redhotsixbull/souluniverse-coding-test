# sql-optimize

소울유니버스 코딩 테스트 Part B 전용. 쿼리 하나를 **진단→개선→인덱스→근거 기록**까지 처리하는 절차.
평가 기준상 "왜 이 쿼리/인덱스가 빠른가"의 **설명**이 결과물 못지않게 중요하다.

## 전제 (지킬 것)
- 작성 위치: `sql/queries_b.sql`의 `[개선된 쿼리]` / `[CREATE INDEX]` 자리에만.
- **상단 스키마 블록과 `[문제 쿼리]`는 수정 금지.**
- 규모 가정: `users`(~50만), `counselors`(~5천), `chat_rooms`(~200만), 인덱스는 PK만.

## 절차

### 1. 진단 (Diagnose)
- 성능 결함을 항목화: 인덱스 미사용(컬럼을 함수로 감쌈 → non-SARGable), 풀스캔, N+1, 불필요 정렬/임시테이블, 결과셋 과잉 컬럼 등.
- "왜 느린가"를 데이터 규모와 연결해 설명 (예: 200만 행 풀스캔).

### 2. 개선 쿼리 (Rewrite)
- 컬럼 가공 제거 → **범위 조건(SARGable)** 으로 전환.
- N+1 → 사전집계 파생테이블 JOIN 등 집합 기반으로.
- 요구 결과셋의 **컬럼·정렬을 정확히** 맞춘다. 불필요 컬럼 제거.
- 요구사항 해석 분기(예: "최근 30일" 경계, 완료 0건 포함 여부)는 택한 해석을 명시.

### 3. 인덱스 (Index)
- `CREATE INDEX` 작성. **복합 인덱스의 컬럼 순서 근거**(등치→범위, 카디널리티, 정렬 활용)를 함께 설명.
- 커버링 인덱스 등 추가 최적화는 쓰기/저장 비용 트레이드오프와 함께 제시.

### 4. 근거 검증 (Justify)
- 가능하면 `EXPLAIN` / `EXPLAIN ANALYZE`로 before→after 비교: `type`(ALL→range/ref), `rows`, `Extra`(Using filesort/temporary 제거 여부).
- 실 DB가 없으면 EXPLAIN 예상과 논리적 근거로 기술하고, 실측이 아님을 밝힌다.

### 5. 기록 (Report)
- `REPORT.md` 해당 섹션 채우기: **문제점 / 개선된 쿼리 / 개선 이유**.
- 개선 이유는 인덱스 설계 근거와 트레이드오프까지. 인덱스 순서가 문제마다 다르면 그 이유도.
