# 04. 의사결정 로그 (Decision Log)

작업하며 내린 결정과 **근거·트레이드오프**를 시간순으로 기록한다.
평가 기준이 "왜 그렇게 결정했는가"이므로, 이 로그가 REPORT.md·커밋 메시지의 근거 원천이 된다.

> 형식: `D-번호 | 결정 | 근거 | 트레이드오프/대안 | 관련 커밋·문서`

---

## D-001. 분석·플랜을 `docs/`로 문서화
- **결정:** 과제 요약/Part A·B 분석/플랜을 `docs/00~03`로 분리 작성.
- **근거:** 평가가 의사결정 과정을 보므로, 분석과 트레이드오프를 남기면 가점. REPORT는 공식 제출물로 별도 유지.
- **대안:** REPORT.md 한 파일에 몰기 → 제출 리포트가 장황해져 가독성 저하. 분리 선택.
- **관련:** `docs/00~03`, 커밋 `docs: ...`

## D-002. Claude 작업 규칙·스킬 범위 = CLAUDE.md + 스킬 2개
- **결정:** 전역 규칙 `CLAUDE.md` 1개 + 스킬 `flutter-bugfix`, `sql-optimize` 2개 작성. 커밋/리포트 전용 스킬은 만들지 않음.
- **근거:** 규칙 준수 자체가 평가 대상 → 규칙 문서가 최우선. 반복 절차(Part A 버그, Part B 쿼리)는 스킬로 형식 강제. 커밋/리포트는 CLAUDE.md 규칙으로 충분(스킬화 시 중복).
- **대안:** 스킬 다수 생성 → 과설계. 최소 구성 선택.
- **관련:** `CLAUDE.md`, `.claude/skills/**`

## D-003. `.claude/` 설정을 제출 repo에 커밋
- **결정:** 규칙·스킬을 `.gitignore`로 숨기지 않고 커밋에 포함.
- **근거:** 체계적 작업 프로세스를 드러내 가점 기대. 평가가 과정을 중시.
- **트레이드오프:** 작은 테스트에 과한 설정으로 보일 위험을 감수. (필요 시 제출 직전 제외 가능)
- **관련:** 커밋 `chore: ...`

## D-004. 커밋 규칙 = Conventional Commits, 한국어, 논리적 단위 분할
- **결정:** Semantic-release 계열 Conventional Commits(`type(scope): subject`) 채택. 메시지는 한국어, *변경 이유*를 본문/제목에 명시. 버그 1개 = 커밋 1개 기본.
- **근거:** README가 "논리적 단위 커밋 + 이유가 드러나는 메시지"를 요구. 일관 규칙으로 평가 가독성↑.
- **대안:** 자유 형식 → 일관성·자동화 약함. Conventional 선택.
- **관련:** `CLAUDE.md` 커밋 규율, 이후 전 커밋.

## D-005. 의사결정을 본 로그에 지속 기록
- **결정:** 작업 중 내리는 결정을 `docs/04-decision-log.md`에 계속 추가하고, 커밋 분할의 근거로 사용.
- **근거:** "왜 그렇게 결정했는가"의 추적성 확보. REPORT.md 작성 시 원천 자료로 재사용.
- **관련:** 본 문서, 이후 전 작업.
